# 01 - 项目概述与架构设计

## 1. 项目背景

MovieMe 是一个前后端分离的电影推荐系统，主要技术目标：

- 掌握 Spring Boot 3.x 全栈开发能力
- 实践 MyBatis-Plus、Redis、JWT 等企业级技术栈
- 手写推荐算法（协同过滤、内容推荐），理解算法原理
- 集成 LLM 大模型 API，实现智能推荐
- 实践爬虫开发、定时任务、反反爬策略
- Docker 容器化部署

### 2.2 架构选型：单体应用 vs 微服务

本项目采用**单体应用 + 模块化包结构**，原因：

| 对比项 | 单体（本项目选择） | 微服务 |
|---|---|---|
| 部署复杂度 | 低，一个 jar 即可 | 高，需要注册中心、网关等 |
| 开发效率 | 高，适合个人项目 | 低，基础设施成本大 |
| 学习价值 | 聚焦业务和算法 | 学习分布式，但偏离核心目标 |
| 面试表达 | "模块化设计，可平滑拆分微服务" | 过度设计，面试官会质疑 |

**关键点**：虽然是单体，但包结构按业务模块划分（`module/user`、`module/movie`...），每个模块有独立的 controller/service/mapper，展示了清晰的边界意识。

## 3. 分层架构详解

### 3.1 请求处理流程

```
HTTP 请求
  → JwtAuthenticationFilter (提取 Token，设置 SecurityContext)
  → Spring Security FilterChain (权限校验)
  → Controller (参数校验 @Valid，调用 Service)
  → Service (业务逻辑，事务管理)
  → Mapper (MyBatis-Plus 操作数据库)
  → 数据库

HTTP 响应
  ← GlobalExceptionHandler (统一异常处理)
  ← Result<T> 统一包装 {code, message, data}
```

### 3.2 各层职责

| 层 | 职责 | 约束 |
|---|---|---|
| **Controller** | 接收请求、参数校验、调用 Service、返回 Result | 不包含业务逻辑，不直接操作数据库 |
| **Service** | 业务逻辑、事务管理、调用 Mapper 和其他 Service | 不依赖 HttpServletRequest 等 Web 层对象 |
| **Mapper** | 数据库 CRUD，MyBatis-Plus BaseMapper | 只做数据访问，不包含业务逻辑 |
| **Entity** | 数据库表的 Java 映射 | 使用 `@TableName`、`@TableId` 注解 |
| **DTO** | 接收前端请求参数 | 使用 `@Valid` 校验注解 |
| **VO** | 返回给前端的数据 | 不暴露敏感字段（如密码） |

### 3.3 DTO / Entity / VO 转换关系

```
前端请求 → DTO (参数校验)
              ↓ 手动转换
           Entity (数据库操作)
              ↓ 手动转换
           VO (响应输出，脱敏)
              ↓
前端响应 ← Result<VO>
```

## 4. 设计模式

### 4.1 策略模式 (Strategy Pattern)

用于推荐引擎和 LLM Provider 选择：

```java
// 推荐策略接口
public interface RecommendStrategy {
    List<MovieVO> recommend(Long userId, int count);
    String getStrategyName(); // "COLLABORATIVE" / "CONTENT_BASED" / "LLM"
}

// 三个实现类
@Service
public class CollaborativeFilteringService implements RecommendStrategy { ... }

@Service
public class ContentBasedService implements RecommendStrategy { ... }

@Service
public class LlmRecommendService implements RecommendStrategy { ... }

// 工厂：根据参数选择策略
@Service
public class RecommendStrategyFactory {
    private final Map<String, RecommendStrategy> strategyMap;

    public RecommendStrategyFactory(List<RecommendStrategy> strategies) {
        this.strategyMap = strategies.stream()
            .collect(Collectors.toMap(RecommendStrategy::getStrategyName, s -> s));
    }

    public RecommendStrategy getStrategy(String name) {
        return strategyMap.get(name);
    }
}
```

**面试考点**：为什么用策略模式？
> 新增推荐算法只需实现接口+注册 Bean，无需修改已有代码，符合开闭原则（OCP）。

### 4.2 模板方法模式 (Template Method)

用于爬虫的数据解析流水线：

```java
public abstract class AbstractMovieParser {
    // 模板方法：定义解析流程
    public final Movie parse(String rawData) {
        Movie movie = new Movie();
        movie.setTitle(parseTitle(rawData));
        movie.setDoubanRating(parseRating(rawData));
        movie.setSummary(parseSummary(rawData));
        // ... 其他字段
        return movie;
    }

    protected abstract String parseTitle(String rawData);
    protected abstract BigDecimal parseRating(String rawData);
    protected abstract String parseSummary(String rawData);
}

// JSON 解析器和 HTML 解析器分别实现抽象方法
```

### 4.3 统一响应 + 全局异常处理

```java
// 统一响应格式
{
    "code": 200,        // 状态码
    "message": "操作成功", // 提示信息
    "data": { ... }     // 业务数据
}

// 全局异常处理：@RestControllerAdvice 捕获所有异常
// BusinessException → 业务错误码
// MethodArgumentNotValidException → 参数校验失败
// Exception → 500 兜底
```

## 5. 数据库设计

### 5.1 ER 关系图

```
users 1───N ratings N───1 movies
```

### 5.2 核心表设计

#### users 用户表

| 字段 | 类型 | 说明 |
|---|---|---|
| id | BIGINT AUTO_INCREMENT | 主键 |
| username | VARCHAR(50) UNIQUE | 用户名 |
| password | VARCHAR(255) | BCrypt 加密密码 |
| email | VARCHAR(100) UNIQUE | 邮箱 |
| nickname | VARCHAR(50) | 昵称 |
| role | TINYINT DEFAULT 0 | 0=用户, 1=管理员 |
| status | TINYINT DEFAULT 1 | 0=禁用, 1=启用 |
| created_at | DATETIME | 创建时间 |
| updated_at | DATETIME | 更新时间 |

#### movies 电影表

| 字段 | 类型 | 说明 |
|---|---|---|
| id | BIGINT AUTO_INCREMENT | 主键 |
| douban_id | VARCHAR(20) UNIQUE | 豆瓣 ID（去重关键） |
| title | VARCHAR(200) | 电影名称 |
| original_title | VARCHAR(200) | 原始名称 |
| poster_url | VARCHAR(500) | 海报 URL |
| year | SMALLINT | 上映年份 |
| douban_rating | DECIMAL(3,1) | 豆瓣评分 |
| douban_votes | INT | 豆瓣评分人数 |
| local_rating | DECIMAL(3,1) | 本站评分（用户评分均值） |
| local_votes | INT | 本站评分人数 |
| summary | TEXT | 剧情简介 |
| country | VARCHAR(100) | 制片国家 |
| duration | VARCHAR(50) | 片长 |
| release_date | DATE | 上映日期 |

**索引设计**：
- `UNIQUE idx_douban_id (douban_id)` — 爬虫去重
- `INDEX idx_year (year)` — 按年份过滤
- `INDEX idx_douban_rating (douban_rating)` — 按评分排序
- `FULLTEXT ft_title_summary (title, summary) WITH PARSER ngram` — 中文全文搜索

#### ratings 评分表

| 字段 | 类型 | 说明 |
|---|---|---|
| id | BIGINT AUTO_INCREMENT | 主键 |
| user_id | BIGINT | 用户 ID（外键） |
| movie_id | BIGINT | 电影 ID（外键） |
| score | TINYINT | 评分 1-10 |
| comment | TEXT | 短评 |
| UNIQUE (user_id, movie_id) | | 每人每部电影只能评一次 |

**面试考点**：为什么用联合唯一索引？
> 数据库层面保证一致性，比应用层 `SELECT + INSERT` 更可靠，避免并发重复评分。

## 7. 安全设计

### 7.1 JWT 认证流程

```
1. 用户登录 → 服务端验证密码 → 生成 accessToken (2h) + refreshToken (7d)
2. refreshToken 存入 Redis (key: auth:refresh_token:{userId})
3. 前端每次请求携带 Header: Authorization: Bearer {accessToken}
4. JwtAuthenticationFilter 解析 Token → 设置 SecurityContext
5. accessToken 过期 → 前端用 refreshToken 调用 /auth/refresh 换新 Token
```

### 7.2 Spring Security 权限配置

```
公开接口：/api/v1/auth/**, GET /api/v1/movies/**, Swagger 文档
认证接口：评分、收藏、推荐、个人信息
管理员接口：/api/v1/admin/** (爬虫控制)
```

### 7.3 密码安全

- BCrypt 加密，strength=12（约 250ms 计算时间，暴力破解不可行）
- 数据库只存 hash，永不存明文

## 8. Redis 缓存设计

| Key 模式 | 用途 | TTL |
|---|---|---|
| `auth:refresh_token:{userId}` | Refresh Token 存储 | 7 天 |
| `movie:detail:{id}` | 电影详情缓存 | 30 分钟 |
| `movie:page:{queryHash}` | 电影列表页缓存 | 10 分钟 |
| `movie:genres` | 类型列表 | 24 小时 |
| `llm:recommend:{userId}:{hash}` | LLM 推荐结果缓存 | 1 小时 |
| `recommend:similarity_matrix` | 协同过滤相似度矩阵 | 7 天（每周重算） |
