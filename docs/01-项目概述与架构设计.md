# 01 - 项目概述与架构设计 (v2)

## 1. 项目定位

MovieMe 是一个**电影浏览 + 个性化推荐**的全栈学习项目，目标是用一个真实业务场景串起 Java 后端 / Vue 前端 / 数据库 / 推荐算法 / 大模型集成等技术。

### 1.1 学习目标
- Spring Boot 3.4 全栈：JWT、MyBatis-Plus、Validation、Scheduling、Async、AOP
- 数据库设计：38 张表的复杂 schema，外键、索引、JSON 字段、FULLTEXT、ENUM
- 爬虫工程：反反爬（UA 轮换、退避、风控页识别）+ 调度
- 推荐系统：Content-Based + Item-Based CF + LLM 三策略融合
- 数据导入：JSON 种子导入 + 幂等 upsert + 关联表 replace 模式

### 1.2 数据基底
- `data/top250.json` 一份预先抓好的豆瓣 Top250（含 50 条/部短评、奖项、相关电影、人物 ID + 头像、5 星分布、Top250 quote 等富字段）
- `data/images/` 250 张已下载的海报
- 后续靠现有爬虫（豆瓣 j/search_subjects + /subject/{id}/）增量补充非 Top250 电影

## 5. 包结构原则

- **按业务模块组织**（`module/user`, `module/movie`），不是按层（`controller/`, `service/` 全局）。
- 每个 module 内部仍按 controller / service / mapper / entity 分子包。
- 跨模块复用的东西放 `common/`。
- 全局基础设施放 `config/`。
- 这种结构便于将来按业务边界拆微服务（虽然本项目永远不会拆）。

## 6. 数据库 v2 概览（详见 03-数据库Schema详解）

38 张表分 8 个域：

```
1) 用户域 (1)         users
2) 字典域 (5)         genres / countries / languages / tags / award_ceremonies
3) 内容主域 (2)       movies / persons
4) 关联表 (7)         movie_genre / country / language / tag / director / writer / actor
5) 富字段域 (8)       movie_top250 / aka / release_date / award /
                     related / comment / rating_dist / genre_rank
6) 用户行为域 (4)     ratings / favorites / view_history / search_history
7) 推荐特征域 (8)     movie_features / user_features / user_genre_pref /
                     user_person_pref / movie_similarity / movie_co_occurrence /
                     user_reco_cache / llm_reco_log
8) 系统日志域 (3)     crawl_logs / recommendation_logs / import_logs
```

设计要点：
- **persons 独立表**：豆瓣人物 ID 唯一，三张关联表 `movie_director / writer / actor` 都引用 `persons.id`，避免人名重复存储。
- **多值字段全部拆关联表**：countries / languages / aka / release_dates / 都是数组，绝不在主表搞 `/` 拼接字符串。
- **推荐特征域厚重**：8 张表专门为推荐准备（电影特征向量、用户偏好、相似度矩阵、共现、缓存），让推荐能"算"出来。
- **外键 + ON DELETE CASCADE**：保证关联数据级联清理。

## 7. 推荐系统的数据流

```
导入 (Phase B)
    ─ top250.json → SeedImportService → 38 表
    ─ 爬虫 → 同样的 schema

离线计算 (Phase C，定时 job)
    ─ MovieFeatureBuilder
        从 movie_genre / movie_actor / movie_director 等聚合
        → movie_features (genre_vector, cast_vector, decade)
    ─ ContentSimilarityCalculator
        movie_features 两两 cosine
        → movie_similarity (algorithm='CONTENT')
    ─ CFItemSimilarityCalculator
        ratings 矩阵 → 余弦
        → movie_similarity (algorithm='CF_ITEM')
    ─ CoOccurrenceCalculator
        ratings + favorites
        → movie_co_occurrence (co_count, pmi)

在线推理 (Phase 后续)
    用户请求 → RecommendationService
        ─ 冷启动 → CONTENT_BASED top-rated 兜底
        ─ 老用户 → HYBRID(content 0.3, cf 0.5, llm 0.2)
    → 写入 user_reco_cache (24h TTL)
    → 返回前端
```

## 9. 关键设计决策（Why?）

### 9.1 为什么 persons 独立表？
top250.json 里同一个人物（比如摩根·弗里曼）出现在多部电影里，每次都带 `douban_person_id`、`avatar`、`profile_url`。如果直接在 `movie_actor` 里冗余存名字，将来想做"摩根·弗里曼演过的所有电影"或"他的演员页"会很丑。独立 persons 表 + 通过 `douban_person_id` 唯一键去重，是标准做法。

### 9.2 为什么短评要单独建表？
- 250 部 × 50 条 = 12500 条豆瓣短评 + 后续站内用户的短评，都需要持久化。
- `source=0/1` 区分豆瓣源 vs 站内源，避免两套表。
- `douban_comment_id UNIQUE` 防重导入。
- 站内用户写短评时 `user_id` 关联 `users.id`，豆瓣源时为 NULL，`author_name/avatar` 直接存。

### 9.3 为什么富字段（rating_dist / genre_rank）值得存？
- 详情页"5 星好评 85.6%"这种数据极大提升页面信息密度。
- 类型百分位"好于 99% 剧情片"是推荐解释（"为什么这部好"）的素材。
- 这些字段是豆瓣慢慢算出来的，我们抓一份白嫖很划算。

### 9.4 为什么推荐特征域占 8 张表？
- **`movie_features`**：每部电影的向量缓存，避免每次推荐都从原始关联表聚合。
- **`movie_similarity`**：top-K 相似预计算，O(1) 查询。
- **`movie_co_occurrence`**："看 A 也看 B" 信号，配合 PMI 比纯 count 更鲁棒。
- **`user_features`**：用户偏好向量。
- **`user_genre_pref` / `user_person_pref`**：细粒度偏好，用于"为什么推这部"解释。
- **`user_reco_cache`**：推荐结果缓存（24h TTL），避免每次请求都重算。
- **`llm_reco_log`**：成本/质量监控。

### 9.5 为什么不用 Redis 存这些推荐数据？
- MySQL 已经够：250 部规模下 movie_similarity 顶多几万行，不到 10 MB。
- JSON 列存 vector 比 Redis 操作更直观。
- 用户量大了再改 Redis 也不晚。**先做对，再做快**。

## 10. 设计模式应用

| 模式 | 在哪用 | 价值 |
|---|---|---|
| 策略模式 | RecommendationService 选 ContentBased/CF/LLM | 算法热切换、新增不改老代码 |
| 模板方法 | DoubanMovieParser 抽象出 list/detail 解析骨架 | 复用流程，子类只填 parse 步骤 |
| 工厂模式 | LLMProvider 选 Gemini/Claude/OpenAI | API 密钥可用就上，否则降级 |
| 责任链 | LLMService 多 Provider 降级 | 第一个挂自动 try 第二个 |
| 单一职责 | 每个 Mapper / Service / Calculator 各司其职 | 容易测试、容易替换 |

## 11. 安全设计

### 11.1 JWT 认证
```
登录 → AccessToken (2h) + RefreshToken (7d)
RefreshToken 存 Redis: auth:refresh_token:{userId}
请求带 Authorization: Bearer {access}
JwtAuthenticationFilter 解析并设置 SecurityContext
```

### 11.2 Spring Security 路由策略
```
公开：/api/v1/auth/**, GET /api/v1/movies/**, swagger
认证：评分 / 收藏 / 推荐 / 浏览历史 / 我的页面
管理员：/api/v1/admin/**（爬虫 + 种子导入手动触发）
```

### 11.3 密码安全
- BCrypt cost=12（约 250ms 计算时间）
- 数据库只存 hash

### 11.4 防滥用
- 登录失败计数（计划：Redis 记 5 次失败 / 15 分钟，触发后短暂锁定）
- 评分 API 限频（计划：每用户每电影最多 1 评，UNIQUE 数据库约束已保证）
