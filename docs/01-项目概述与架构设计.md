# 01 - 项目概述与架构设计

## 1. 项目定位

MovieMe 是一个**电影浏览 + 个性化推荐**的全栈学习项目，把 Java 后端 / Vue 前端 / 数据库 / 推荐算法 / 大模型集成串成一个真实业务。

### 1.1 学习目标
- Spring Boot 3.4 全栈：JWT、Spring Security、MyBatis-Plus、Validation、Scheduling、AOP、文件上传
- 数据库设计：40+ 张表的复杂 schema，外键、索引、JSON 字段、FULLTEXT、ENUM
- 爬虫工程：Java 调度 + Python 执行的混合架构，反反爬（UA 轮换、退避、风控页识别）
- 推荐系统：Content-Based + Item-Based CF + LLM 三策略融合（默认关闭）
- 数据导入：JSON 种子导入 + 幂等 upsert + 关联表 replace 模式
- 前端工程：Vue 3 Composition API + Pinia + Tailwind 4 + 自定义指令 + 设计 token

### 1.2 数据基底
- `data/top250.json` —— 一份预先抓好的豆瓣 Top250（含每部 50 条短评、奖项、相关电影、人物 ID + 头像、5 星分布、Top250 quote 等富字段）
- `data/images/` —— 250 张已下载的海报
- `crawler/` —— Python 脚本（榜单/详情/年度/补抓），由后端 `CrawlerScheduler` 通过 4 个 @Scheduled 定时调度

## 5. 包结构原则

- **按业务模块组织**（`module/user`, `module/movie`），不是按层（`controller/`, `service/` 全局）。
- 每个 module 内部仍按 controller / service / mapper / entity / dto / vo 分子包。
- 跨模块复用的东西放 `common/`。
- 全局基础设施放 `config/`。
- 这种结构便于将来按业务边界拆微服务（虽然本项目永远不会拆）。

## 6. 数据库 v2 概览（详见 [03-数据库Schema详解](./03-数据库Schema详解.md)）

40+ 张表分 8 个域：

| 域 | 张数 | 代表表 |
|---|---|---|
| 用户域 | 1 | users |
| 字典域 | 5 | genres / countries / languages / tags / award_ceremonies |
| 内容主域 | 2 | movies / persons |
| 关联表域 | 7 | movie_genre / country / language / tag / director / writer / actor |
| 富字段域 | 10 | movie_top250 / aka / release_date / award / related / play_link / comment / comment_vote / rating_dist / genre_rank |
| 用户行为域 | 4 | ratings / favorites / view_history / search_history |
| 推荐特征域 | 8 | movie_features / user_features / user_genre_pref / user_person_pref / movie_similarity / movie_co_occurrence / user_reco_cache / llm_reco_log |
| 系统日志域 | 3 | crawl_logs / recommendation_logs / import_logs |

设计要点：
- **persons 独立表**：豆瓣人物 ID 唯一，三张关联表 `movie_director / writer / actor` 都引用 `persons.id`，避免人名重复存储。
- **多值字段全部拆关联表**：countries / languages / aka / release_dates / 等绝不在主表做字符串拼接。
- **推荐特征域厚重**：8 张表专门为推荐准备，让推荐能"算"出来。
- **评论双源**：`movie_comment.source = 0` 豆瓣抓取、`= 1` 站内用户；`comment_vote` 单独表存站内点赞。
- **外键 + ON DELETE CASCADE**：保证关联数据级联清理。

## 7. 推荐系统数据流（默认关闭）

```
导入 (Phase B，已完成)
    top250.json → SeedImportService → 38+ 表
    Python 爬虫  → 同样的 schema 增量补充

离线计算 (Phase C，已就位，默认 enabled=false)
    MovieFeatureBuilder    → movie_features (genre_vector / cast_vector / decade)
    ContentSimilarityCalc  → movie_similarity (algorithm='CONTENT')
    CFItemSimilarityCalc   → movie_similarity (algorithm='CF_ITEM')
    CoOccurrenceCalc       → movie_co_occurrence (co_count, pmi)
    UserFeatureBuilder     → user_features + user_genre_pref + user_person_pref
    Popularity/Quality/Freshness Scorer → movies.*_score

在线推理 (Phase E 路线图，未实现)
    用户请求 → RecommendationService
        冷启动 → CONTENT_BASED top-rated 兜底
        老用户 → HYBRID(content 0.3, cf 0.5, llm 0.2)
    → 写入 user_reco_cache (24h TTL)
    → 返回前端
```

启用方式：把 `application.yml` 的 `moveme.recommendation.enabled` 改为 `true`，并按需要把 `bootstrap-on-start` 也改为 `true`。详见 [41-推荐特征计算](./41-推荐特征计算.md)。

## 8. 关键设计决策（Why?）

### 8.1 persons 为什么独立表
top250.json 里同一个人物（如摩根·弗里曼）出现在多部电影里，每次都带 `douban_person_id`、`avatar`、`profile_url`。如果直接在 `movie_actor` 里冗余存名字，将来做"演员页"或"用户对人物的偏好"会很丑。独立 persons 表 + `douban_person_id` UNIQUE 去重是标准做法。

### 8.2 评论为什么单源表
- 250 部 × 50 条豆瓣短评 + 后续站内用户短评，都需要持久化。
- `source=0/1` 区分来源，避免两套表两套查询。
- `douban_comment_id UNIQUE` 防重导。
- 站内点赞独立表 `comment_vote(comment_id, user_id, ...)` 而非 `comment.votes` 列：站内用户能取消点赞需要明细记录。

### 8.3 富字段（rating_dist / genre_rank）为什么单独存
- 详情页"力荐 85.6%"这种数据极大提升信息密度。
- "好于 99% 剧情片" 是推荐解释（"为什么这部好"）的素材。
- 这些数据从豆瓣慢慢算出来，我们抓一份白嫖很划算。

### 8.4 推荐特征为什么默认关闭
- 250 部规模数据集下，没有用户行为时 CF 矩阵全空，跑 CF Calculator 没意义。
- Phase C 框架已就位，是为未来用户基数起来后能"打开就用"准备的。
- 启用条件：站内 ≥ 100 个活跃用户、≥ 1000 条评分，再打开 `enabled=true`。

### 8.5 海报为什么本地化
- 豆瓣海报 URL 有 Referer 防盗链，浏览器 `<img>` 直接加载会 403。
- 导入阶段 `PosterFileImporter` 把 `data/images/*.jpg` 拷到 `data/posters/`，对外通过 `/static/posters/` 暴露。
- 前端 `resolvePoster()` 优先用 `posterLocalPath`，降级到 `posterUrl`。

### 8.6 爬虫为什么 Java 调度 + Python 执行
- Java 端管事务/日志/调度比 Python 顺手（`CrawlerScheduler` + `CrawlLog`）。
- 豆瓣爬取细节（HTML 解析 / 反反爬）Python 生态更成熟（`requests` + `BeautifulSoup`）。
- Java 通过 `commons-io` 的 `ProcessBuilder` 拉起 Python，stdout 落库为 `CrawlLog`。

## 9. 设计模式应用

| 模式 | 在哪用 | 价值 |
|---|---|---|
| 单一职责 | 每个 Mapper / Service / Calculator 各司其职 | 容易测试、容易替换 |
| 策略模式 | RecommendationStrategy（ColdStart / Content / CF / Hybrid） | 算法热切换 |
| 模板方法 | SeedImportService 字段映射骨架 | 子流程可独立测 |
| 工厂模式 | LLMProvider 选 Gemini / Claude / OpenAI（按 key 可用降级） | 后续扩展自然 |
| 适配器 | resolvePoster() 远程 URL ↔ 本地路径 | 防盗链兜底 |

## 10. 安全设计

### 10.1 JWT 认证
```
登录 → AccessToken (2h) + RefreshToken (7d)
RefreshToken 存 Redis: auth:refresh_token:{userId}
请求带 Authorization: Bearer {access}
JwtAuthenticationFilter 解析并设置 SecurityContext
```

### 10.2 Spring Security 路由策略（`SecurityConfig`）

| 路径 | 策略 |
|---|---|
| `/api/v1/auth/**` | permitAll |
| `GET /api/v1/movies/**` | permitAll |
| `GET /api/v1/persons/**` | permitAll |
| `/static/**` (GET/HEAD) | permitAll |
| `/doc.html` / `/webjars/**` / `/v3/api-docs/**` / `/swagger-ui/**` | permitAll |
| `/api/v1/admin/**` | hasRole("ADMIN") |
| 其余 | 需认证 |

### 10.3 密码安全
- BCrypt cost=12（约 250ms 计算时间）
- 数据库只存 hash

### 10.4 CORS
- `WebMvcConfig.addCorsMappings("/api/**")` 允许 `localhost:5173` 等前端来源
- `allowCredentials(true)` 配合 JWT 拦截

### 10.5 输入校验
- DTO 上加 `@Valid` + `@NotBlank` / `@Min` / `@Max` / `@Size`（如 `MovieController.searchMovies` 的 q/page/size）
- 全局异常处理器统一返回 `Result.error(400, ...)`
