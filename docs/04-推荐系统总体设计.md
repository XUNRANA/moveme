# 04 - 推荐系统总体设计

> **实施状态**：Phase A 数据库 8 张推荐特征表已落地；Phase B 种子数据已导入；Phase C Calculator 框架（特征 / 相似度 / 共现）已实现，但 `application.yml` 中 `moveme.recommendation.enabled = false` 默认关闭。在线推理（RecommendationController/Service）属于 Phase E，待用户基数与评分密度上来后再启用。**Phase F LLM 推荐已上线**（MiMO 大模型，聊天 + 一键推荐），详见 [34-AI推荐功能说明](./34-AI推荐功能说明.md)。
>
> MovieMe 用 **三策略融合（Content-Based + Item-Based CF + LLM）** + **冷启动兜底** + **24h 推荐缓存**。
>
> 这份文档讲架构、数据流、策略选择、冷启动、缓存。算法细节看 [05 - 推荐算法详解](./05-推荐算法详解.md)；模块结构与如何启用看 [41-推荐特征计算](./41-推荐特征计算.md)。

---

## 1. 设计目标

1. **冷启动可用**：新用户没行为时也要给得出"看着像样"的推荐。
2. **越用越准**：行为越多，推荐越个性化。
3. **可解释**："为什么推这部" 必须能说清楚。
4. **可降级**：LLM 挂了不影响主流程；CF 数据稀疏时回落 Content。
5. **可缓存**：同一用户 24h 内不重复算。

## 3. 三种推荐策略

### 3.1 Content-Based（内容相似）
**思路**：根据电影自身特征（类型/演员/导演/年代/国家）算相似度，给用户看过的电影推相似的。

**优点**：冷启动友好（不需要其他用户数据），可解释（"因为你看过《肖申克》，给你推《阿甘正传》"）。
**缺点**：容易陷入"信息茧房"（推的都很像）。

**何时用**：用户行为少（< 5 次评分/收藏），主要走这条。

**数据来源**：`movie_similarity (algorithm='CONTENT')` —— 离线预计算 250×top50。

### 3.2 Item-Based CF（协同过滤）
**思路**：根据"看过 A 的人也看过 B" 的用户行为模式给推荐。

**优点**：能发现非显性关联（"喜欢《盗梦空间》的人也喜欢《让子弹飞》"，类型完全不同）。
**缺点**：冷启动废，需要足够评分密度。

**何时用**：站内有 ≥ 5 个共同评分用户的电影对足够多时，进入主推。

**数据来源**：`movie_similarity (algorithm='CF_ITEM')` + `movie_co_occurrence`。

### 3.3 LLM（大模型）
**思路**：把用户偏好画像 + 候选电影列表喂给 LLM，让它推理"哪些电影最适合这个用户"，并给出自然语言解释。

**优点**：理由有人味；能跨越内容/CF 的局限。
**缺点**：贵（每次调用几美分）+ 慢（2-5 秒）+ 不稳（可能推不在库的电影）。

**何时用**：补充策略（占融合权重 0.2）；或用户主动点"换一批" / "用 AI 推荐"按钮才触发。

**Provider 链**：Gemini > Claude > OpenAI（按 API key 是否配置依次降级）。

## 4. 冷启动策略

| 用户状态 | 策略 | 推荐池 |
|---|---|---|
| **未登录** | 热门 + 高分 | `movies` 按 `popularity_score DESC, quality_score DESC` top-N |
| **登录但 0 行为** | Top250 高分 + 类型多样性采样 | `movie_top250` 取 rank ≤ 50，按 5 个主流类型各取若干 |
| **行为 < 5** | Content-Based + 兜底（弱混合） | 80% Content + 20% 热门 |
| **行为 ≥ 5** | HYBRID 全策略融合 | 见下节 |

冷启动池缓存 6 小时（不是用户特定，可全局共享）。

## 5. HYBRID 融合（老用户主路径）

```
final_score(user, movie) =
    0.30 * normalize(content_score)     // 内容相似度（基于用户已看过电影）
  + 0.50 * normalize(cf_score)          // 协同过滤分
  + 0.20 * normalize(llm_score)         // LLM 排序分（可选，开关控制）
  + 0.10 * popularity_bonus             // 长尾防过冷
  - 1000 * has_seen                     // 已看过的硬过滤
```

**归一化**：每个 score 做 min-max 到 [0,1]，避免量级失衡。

**多样性兜底**：top-30 候选里强制至少覆盖 3 个不同主类型，避免"全是悬疑"。

最终输出：
```json
{
  "items": [
    {
      "movie_id": 42,
      "title": "盗梦空间",
      "score": 0.87,
      "reason": "因为你看过《星际穿越》和《信条》",
      "strategy_breakdown": {"content": 0.65, "cf": 0.92, "llm": 0.78}
    }
  ],
  "strategy": "HYBRID",
  "cached_at": "2026-04-28T14:00:00",
  "expires_at": "2026-04-29T14:00:00"
}
```

## 7. 数据流（离线 vs 在线）

### 7.1 离线（凌晨 job，几秒到几分钟）

```
movies/persons/movie_*关联 ─┐
                            ├──▶ MovieFeatureBuilder ──▶ movie_features
ratings/favorites ─────────┘                              │
                                                          ▼
                                ContentSimilarityCalculator ──▶ movie_similarity (CONTENT)

ratings/favorites ──▶ CFItemSimilarityCalculator ──▶ movie_similarity (CF_ITEM)
                  └─▶ CoOccurrenceCalculator     ──▶ movie_co_occurrence

ratings/favorites/view_history ──▶ UserFeatureBuilder ──▶ user_features
                                                          │
                                                          ▼
                                                       user_genre_pref + user_person_pref

movies + douban_votes/rating + release_date ──▶ Scorer ──▶ movies.popularity/quality/freshness
```

### 7.2 在线（用户请求，目标 < 200ms）

```
user_id → user_features(JSON) + user_genre_pref(top 5)
       → 拼推荐池: 用户已评分电影的 top-K 相似（content + cf）+ 类型偏好热门
       → 排序融合
       → 过滤已看 + 多样性约束
       → 写 user_reco_cache
       → 返回 movies metadata
```

## 8. 缓存策略

| 缓存层 | 内容 | TTL | 失效策略 |
|---|---|---|---|
| **user_reco_cache (MySQL)** | 用户个性化推荐结果 | 24h | 用户写新评分/收藏 → 主动失效 |
| **冷启动池 (Redis)** | 全局 top-N 热门高分 | 6h | 定时刷新 |
| **movie metadata (Redis)** | 单部电影详情 JSON | 1h | 后台爬虫更新时主动 evict |

主动失效（关键）：用户行为变了，缓存里的推荐就失效了：

```java
@EventListener(RatingChangedEvent.class)
public void invalidate(RatingChangedEvent e) {
    userRecoCacheMapper.deleteByUserId(e.getUserId());
}
```

## 9. 解释生成（"为什么推这部"）

每条推荐结果都要带 reason。来源：

| 策略 | reason 模板 |
|---|---|
| Content-Based | "因为你看过《{seedMovie}》和它相似（共享类型：{commonGenres}，主演：{commonActors}）" |
| Item-CF | "看过《{seedMovie}》的用户也很喜欢这部" |
| Genre Pref | "你最爱的{genre}类型 / 评分 {rating}" |
| LLM | LLM 自己给的自然语言解释 |
| Hybrid | 取分数最高的子策略的 reason |

## 10. 评估指标（后续）

- **CTR**（推荐点击率）：reco_view → movie_detail click
- **Coverage**（推荐覆盖率）：被推过的不同电影 / 总电影数
- **Diversity**（多样性）：推荐 list 内类型熵
- **Long-tail rate**：推荐里非 top250 电影占比
